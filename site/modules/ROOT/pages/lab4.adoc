= Lab 4: Graph-Enriched Search

In Lab 3 you built an agent that calls simple tools. Now you connect an agent to your knowledge graph and explore what makes searching a knowledge graph different from searching a standalone vector database. This lab walks through three retrieval strategies that grow in sophistication — from basic vector search, to graph-enriched search that follows relationships to connected entities, to hybrid search that combines semantic and keyword matching.

TIP: Ready to start? Jump to the xref:lab4-instructions.adoc[Lab 4 Instructions] for step-by-step notebook walkthroughs.

.**At a Glance**

* **Vector search** — find text passages whose meaning is closest to your question, even when the exact words differ
* **Graph-enriched search** — after finding matching passages, follow graph relationships to pull in the companies, products, and risk factors connected to them
* **Hybrid search** — combine keyword matching with semantic similarity for broader recall

== From Filings to Searchable Knowledge

Before diving into the retrieval strategies, here is how the data in this lab was prepared and why each step matters.

. **The raw data is SEC 10-K filings.** These are annual financial reports that public companies file with the Securities and Exchange Commission. A single filing can run tens of thousands of words covering business operations, risk factors, financial results, and more.

. **Filings are split into chunks.** A full filing is too long to search precisely, so each one is split into smaller passages called chunks. Each chunk becomes a node in the knowledge graph, linked to its source document and to adjacent chunks so the original reading order is preserved.

. **Each chunk is encoded as an embedding.** An embedding model reads the chunk's text and produces a numerical vector that captures what the text _means_, not just what words it contains. "Supply chain disruption" and "logistics bottleneck risk" produce similar vectors because they describe related concepts, even though they share no keywords.

. **Vector search finds chunks by meaning.** Given a natural-language question, the system encodes it using the same embedding model and searches for stored chunk vectors whose meaning is closest. This is semantic similarity search: it matches on concepts rather than exact words.

. **Vector search returns isolated passages.** The results are ranked text fragments with no surrounding context. You get the chunk text and a similarity score, but nothing about which company filed it, what products it mentions, or what risks it describes.

. **Graph-enriched search follows relationships to fill that gap.** The knowledge graph connects chunks to documents, companies, products, and risk factors through typed relationships. After vector search finds relevant chunks, graph traversal follows those relationships to pull in structured context. The result is not just "here's a matching passage" but "here's a matching passage from Apple's 2024 10-K, mentioning the iPhone, linked to a cybersecurity risk factor."

This lab starts with step 4 (vector search) and builds toward step 6 (graph-enriched search). The chunking, embedding, and graph construction have already been done for you.

The graph you explored in Lab 1 — companies, products, risk factors, asset managers — is the structured layer. The documents and chunks described above are the unstructured layer, added on top. The `FILED` relationship bridges them: `(Company)-[:FILED]->(Document)`. When graph-enriched search traverses from a chunk to its source document to the company that filed it, it crosses from unstructured text into the structured entities you already know. That traversal chain, `(Chunk)-[:FROM_DOCUMENT]->(Document)<-[:FILED]-(Company)`, is what makes graph-enriched search possible.

== Three Retrieval Strategies

Each notebook builds a retrieval strategy from scratch. You write the Cypher queries by hand inside `@tool` functions, manage embedding generation yourself, and wire everything through MCP. By the end of this lab you will understand exactly what vector search, graph traversal, and fulltext search look like at the query level — the mechanics that the neo4j-graphrag library abstracts away in xref:lab5.adoc[Lab 5].

Each notebook returns more context than the previous one. Vector search returns chunk text. Graph-enriched search returns chunk text plus connected entities and document metadata. Hybrid search returns results from both keyword and semantic matching, giving the agent the broadest possible context to reason over.

=== Vector Search

Notebook 01 introduces the basic pattern. A custom `@tool` function embeds the natural-language query using Bedrock Nova and executes the vector search Cypher through MCP. The vector index (`chunkEmbeddings`) finds the chunks whose embeddings are most similar to the query embedding. The results are isolated chunks ranked by similarity — no graph context yet.

=== Graph-Enriched Search

Notebook 02 extends vector search by adding graph traversal. After finding semantically similar chunks, the agent follows relationships to pull in document metadata, neighboring chunks (via `NEXT_CHUNK`), and connected entities — companies, products, and risk factors. This context enrichment gives the agent more information to work with when generating answers.

=== Hybrid Search

Notebook 03 introduces an alternative search method: fulltext keyword search using Lucene operators against the `search_chunks` and `search_entities` indexes. It then combines both approaches for agent-driven hybrid search using custom `@tool` wrappers that encapsulate embedding generation.

== How the Agent Connects: MCP

The agent communicates with Neo4j through MCP (Model Context Protocol), an open standard for agent-to-tool communication. The agent sends Cypher through an `execute-query` tool exposed by the Neo4j MCP Server. The server handles the connection, runs the query, and returns structured results. MCP is the transport layer for these queries, not the retrieval strategy itself. For the Text2Cypher pattern — where the agent discovers the schema and writes its own Cypher — see xref:lab6.adoc[Lab 6].

NOTE: This lab uses the https://github.com/neo4j/mcp[Neo4j MCP Server], an official MCP server that exposes a Neo4j database as a set of tools an agent can call — including `execute-query` for running Cypher, schema introspection, and index management. The server has been deployed to AgentCore using the https://github.com/neo4j-partners/aws-starter/tree/main/neo4j-agentcore-mcp-server[neo4j-agentcore-mcp-server starter] and connects to a database pre-loaded with Document and Chunk nodes, vector embeddings (`chunkEmbeddings` index), and fulltext indexes (`search_chunks`, `search_entities`). The workshop admin set this up before the workshop. You do not need to deploy the MCP server or create these data structures yourself — everything is ready to query.

== The @tool Pattern

Notebooks 01 and 03 wrap embedding generation and Cypher execution inside custom `@tool` functions. The agent receives a natural-language question, decides which search to run, and calls a tool function. That function handles the mechanical work the LLM should never see: generating the embedding, building the Cypher query, and sending it to Neo4j through the MCP connection. The results come back as text that the agent can reason over and present to the user.

The important design choice is where the embedding lives. A 1024-dimensional vector is roughly 8–10KB of floating-point numbers. If that array passes through the LLM's context window, it wastes tokens on data the model cannot interpret, doubles the payload (once in the message, once in the tool call), and risks the model silently truncating or rounding values. Wrapping embedding generation inside a `@tool` function keeps the vector on the data plane. The agent calls `vector_search_tool("supply chain risks")` and receives chunk text and scores. It never touches the float array.

This is the same separation of concerns that the neo4j-graphrag library enforces in Lab 5: the `VectorRetriever` generates embeddings and queries the index internally, and the LLM only sees retrieved text. The `@tool` wrapper achieves the same boundary when working through MCP instead of a direct driver connection.
