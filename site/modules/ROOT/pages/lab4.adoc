== Chunks, Embeddings, and Graph-Enriched Search

Most documents are too long for an LLM to process in a single pass. Retrieval-Augmented Generation (RAG) solves this by finding the most relevant pieces of a document and feeding only those to the model. The building blocks are:

**Chunks.** A long document is split into smaller passages called chunks. Each chunk is short enough to embed and retrieve individually, but long enough to carry useful context. Chunks are stored as nodes in the graph and linked in sequence so the original reading order is preserved.

**Embeddings.** An embedding model reads a chunk's text and produces a numerical vector that captures what the text _means_, not just what words it contains. "Supply chain disruption" and "logistics bottleneck risk" produce similar vectors because they describe related concepts, even though they share no keywords.

**Vector search.** Given a natural-language question, the system encodes it with the same embedding model and finds stored chunk vectors whose meaning is closest. This is semantic similarity search — it matches on concepts rather than exact words. The results are ranked text fragments with similarity scores.

**The limitation of vector search alone.** Vector search returns isolated passages. You get the chunk text and a score, but nothing about where the passage came from, what entities it mentions, or how it relates to other information in the corpus.

**Graph-enriched search.** A knowledge graph connects chunks to structured entities — documents, companies, products, risk factors — through typed relationships. After vector search finds relevant chunks, graph traversal follows those relationships to pull in structured context. The result is not just "here's a matching passage" but "here's a matching passage, filed by this company, mentioning these products, linked to these risk factors." That additional context grounds the LLM's answer in verified, structured data.

=== Example: SEC 10-K Filings

This lab applies the pattern above to SEC 10-K filings — annual financial reports that public companies file with the Securities and Exchange Commission. A single filing can run tens of thousands of words covering business operations, risk factors, financial results, and more.

. Each filing is split into chunks and stored as `(:Chunk)` nodes linked by `NEXT_CHUNK` relationships.
. Each chunk is encoded into a 1024-dimensional embedding vector using Amazon Bedrock Nova.
. A vector index (`chunkEmbeddings`) enables semantic similarity search across all chunks.
. Graph relationships connect chunks back to their source `(:Document)`, which is linked to the `(:Company)` that filed it, and cross-links (`FROM_CHUNK`) tie chunks to the `(:Product)` and `(:RiskFactor)` entities they mention.

When you ask "What cybersecurity risks does Apple face?", vector search finds the most relevant passages. Graph-enriched search then follows relationships from those passages to return Apple as the filer, the specific products mentioned, and the risk factor classifications — all in a single retrieval step.

== The Two-Layer Graph

The graph you explored in Lab 1 — companies, products, risk factors, asset managers — is the **structured layer**. Notebook 01 adds the **unstructured layer**: Document and Chunk nodes with embeddings, linked by `FROM_DOCUMENT` and `NEXT_CHUNK` relationships. Cross-links (`FROM_CHUNK`) connect entities like Products and RiskFactors to the Chunks that mention them.

The `FILED` relationship bridges the two layers: `(Company)-[:FILED]->(Document)`. When graph-enriched search traverses from a chunk to its source document to the company that filed it, it crosses from unstructured text into the structured entities you already know. That traversal chain — `(Chunk)-[:FROM_DOCUMENT]->(Document)<-[:FILED]-(Company)` — is what makes GraphRAG possible.

[source]
----
Structured Layer (Lab 1)          Unstructured Layer (Lab 4 Notebook 01)
┌──────────────────────┐          ┌──────────────────────────────────────┐
│ (:Company)           │──FILED──▶│ (:Document)◀─FROM_DOCUMENT─(:Chunk) │
│   ├─OFFERS─▶Product  │          │                 │                   │
│   ├─FACES_RISK─▶Risk │          │            NEXT_CHUNK               │
│   └─COMPETES_WITH    │          │                 ▼                   │
│ (:AssetManager)      │          │             (:Chunk)                │
│   └─OWNS─▶Company   │          └──────────────────────────────────────┘
└──────────────────────┘
         ▲                                    │
         └────── FROM_CHUNK ──────────────────┘
         (Products, RiskFactors linked to Chunks)
----

== Retrieval Strategies

=== VectorRetriever

Notebook 02 introduces the `VectorRetriever` from the neo4j-graphrag library. It wraps the vector index and handles embedding the query, running the similarity search, and formatting the results. Connected to an LLM via `GraphRAG`, it builds a complete question-answering pipeline. The results are chunk text ranked by cosine similarity — isolated passages with no graph context.

=== VectorCypherRetriever

Notebook 03 extends vector search with the `VectorCypherRetriever`. After the vector index finds matching chunks, a custom Cypher retrieval query runs on each match to traverse the graph and pull in structured context:

[source,cypher]
----
MATCH (node)-[:FROM_DOCUMENT]->(doc:Document)
OPTIONAL MATCH (doc)<-[:FILED]-(company:Company)
OPTIONAL MATCH (company)-[:FACES_RISK]->(risk:RiskFactor)
OPTIONAL MATCH (product:Product)-[:FROM_CHUNK]->(node)
----

The LLM now receives the matched chunk alongside the filing accession number, company name, product list, and risk factor labels — producing answers grounded in both unstructured text and structured knowledge graph relationships.

== What the Library Does Under the Hood

The neo4j-graphrag retrievers handle several steps internally so you can focus on the query rather than the plumbing. When you call a retriever, it:

1. **Embeds your question** — sends the text to the embedding model (Bedrock Nova) and gets back a 1024-dimensional vector.
2. **Queries the vector index** — runs a Cypher query against the `chunkEmbeddings` index using that vector.
3. **Traverses the graph** (Cypher retrievers only) — executes the `retrieval_query` you provide to follow relationships from matched chunks to connected entities.
4. **Formats results for the LLM** — packages the retrieved text and metadata into a prompt-ready format.

In xref:lab5.adoc[Lab 5], you will do each of these steps yourself through MCP — writing the Cypher by hand, managing embeddings in `@tool` functions, and wiring the results back to the agent. Understanding what the library automates here will make the manual approach in Lab 5 clearer.
