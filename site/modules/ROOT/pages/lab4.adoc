== What Is MCP Retrieval?

Instead of connecting to Neo4j with a driver and using retriever classes directly, an AI agent uses MCP tools to execute Cypher queries against the database. The agent sends Cypher through an `execute-query` tool exposed by the Neo4j MCP Server. The server handles the connection, runs the query, and returns structured results that the agent can reason over.

MCP (Model Context Protocol) is an open standard for agent-to-tool communication. It defines how an agent discovers available tools, calls them with structured inputs, and receives structured outputs. In this lab you use it as a transport layer for Cypher queries. For a deeper look at how MCP works and how to build full MCP agents, see xref:lab6.adoc[Lab 6].

NOTE: This lab uses the https://github.com/neo4j/mcp[Neo4j MCP Server], an official MCP server that exposes a Neo4j database as a set of tools an agent can call — including `execute-query` for running Cypher, schema introspection, and index management. The server has been deployed to AgentCore using the https://github.com/neo4j-partners/aws-starter/tree/main/neo4j-agentcore-mcp-server[neo4j-agentcore-mcp-server starter] and connects to a database pre-loaded with Document and Chunk nodes, vector embeddings (`chunkEmbeddings` index), and fulltext indexes (`search_chunks`, `search_entities`). The workshop admin set this up before the workshop. You do not need to deploy the MCP server or create these data structures yourself — everything is ready to query.

== Three Retrieval Strategies

The three notebooks progress through retrieval strategies that build on each other. Each strategy returns more context than the previous one. Vector search returns chunk text. Graph-enriched search returns chunk text plus connected entities and document metadata. Hybrid search returns results from both keyword and semantic matching, giving the agent the broadest possible context to reason over.

=== Vector Search

Notebook 01 introduces the basic pattern. You embed a natural-language query using Bedrock Nova, pass the embedding to the agent, and let the agent execute vector search Cypher through MCP. The vector index (`chunkEmbeddings`) finds the chunks whose embeddings are most similar to the query embedding.

=== Graph-Enriched Search

Notebook 02 extends vector search by adding graph traversal. After finding semantically similar chunks, the agent follows relationships to pull in document metadata, neighboring chunks (via NEXT_CHUNK), and connected entities — companies, products, and risk factors. This context enrichment gives the agent more information to work with when generating answers.

=== Hybrid Search

Notebook 03 introduces an alternative search method: fulltext keyword search using Lucene operators against the `search_chunks` and `search_entities` indexes. It then combines both approaches for agent-driven hybrid search using custom `@tool` wrappers that encapsulate embedding generation.

=== How the Tool Pattern Works

All three strategies share the same underlying flow. The agent receives a natural-language question, decides which search to run, and calls a custom tool function. That tool function handles the mechanical work the LLM should never see: generating the embedding, building the Cypher query, and sending it to Neo4j through the MCP connection. The results come back as text that the agent can reason over and present to the user.

The important design choice is where the embedding lives. A 1024-dimensional vector is roughly 8–10KB of floating-point numbers. If that array passes through the LLM's context window, it wastes tokens on data the model cannot interpret, doubles the payload (once in the message, once in the tool call), and risks the model silently truncating or rounding values. Wrapping embedding generation inside a `@tool` function keeps the vector on the data plane. The agent calls `vector_search_tool("supply chain risks")` and receives chunk text and scores. It never touches the float array.

This is the same separation of concerns that the neo4j-graphrag library enforces in Lab 5: the `VectorRetriever` generates embeddings and queries the index internally, and the LLM only sees retrieved text. The `@tool` wrapper achieves the same boundary when working through MCP instead of a direct driver connection.

== MCP vs Direct Library

All three notebooks use the same underlying Neo4j indexes and graph structure as Lab 5. The difference is in how the search is executed.

[cols="1,2,2"]
|===
| Aspect | Lab 4 (MCP) | Lab 5 (Library)

| **Connection**
| MCP server via AgentCore Gateway
| Direct Neo4j driver

| **Query execution**
| Agent calls `execute-query` MCP tool
| Retriever class methods

| **Embedding**
| Client-side (Bedrock Nova), passed to agent or wrapped in `@tool`
| Retriever calls embedder internally

| **Hybrid search**
| Agent runs both queries and combines results
| `HybridRetriever` handles score normalization and alpha tuning

| **Graph traversal**
| Cypher in agent prompt
| `retrieval_query` parameter on retriever classes
|===
