= Neo4j Graph Intelligence Platform

A GraphRAG agent needs more than a database. It needs a knowledge layer that stores entities and their relationships, indexes text for vector and keyword retrieval, and exposes query capabilities to external tools. Neo4j's Graph Intelligence Platform provides that knowledge layer across three tiers: a graph database and analytics engine at the foundation, developer and exploration tools in the middle, and graph-powered AI services at the top.

.**At a Glance**

* **Database and Analytics** — a native graph database (AuraDB) for storing and querying the knowledge graph, plus Graph Data Science (GDS) for running graph algorithms like community detection and similarity scoring
* **AI-Powered Graph Tools** — Browser for Cypher queries, Bloom for visual exploration, NeoDash for dashboards, Data Import for loading, and a GraphQL API
* **Graph-Powered AI** — Document Intelligence, Aura Agents, Ontology modeling, Agentic Brain for agent memory, and MCP Server connectivity that turn the knowledge graph into a reasoning substrate for AI agents

== Database and Analytics

The foundation of the platform is a native graph database. Unlike relational databases that compute relationships at query time through joins, Neo4j stores relationships as first-class structures on disk. A traversal from a Company node through its FACES_RISK relationships to connected RiskFactor nodes follows direct pointers rather than scanning join tables. For the multi-hop queries that GraphRAG depends on (find a chunk, traverse to its parent document, then to the company that filed it, then to that company's competitors), this index-free adjacency keeps latency constant regardless of total dataset size.

=== AuraDB

Neo4j AuraDB is the fully managed cloud database used throughout this workshop. It handles provisioning, backups, and scaling, so the labs focus on graph modeling and querying rather than infrastructure. AuraDB supports three capabilities the workshop relies on:

**Vector indexes** store 1024-dimensional embeddings on Chunk nodes. When a user asks a question, the query embedding is compared against the `chunkEmbeddings` index using cosine similarity, and the closest chunks are returned. Labs 4 and 5 use this for semantic search over SEC 10-K filing text.

**Fulltext indexes** provide keyword search. The `search_chunks` index enables BM25-style text matching on Chunk content, and `search_entities` covers Company, Product, and RiskFactor names. Lab 4 combines fulltext results with vector results for hybrid retrieval.

**Cypher** is Neo4j's graph query language. It uses ASCII-art syntax to express patterns: `(company)-[:FACES_RISK]->(risk)` reads like a diagram. Every lab from Lab 1 onward uses Cypher, whether typed directly in the console, embedded in Python, or generated by an LLM in Lab 6's Text2Cypher pattern.

=== Graph Data Science (GDS)

The Graph Data Science library provides over 65 graph algorithms that run on in-memory projections of the database. Algorithms are grouped into categories: centrality (which nodes are most important), community detection (which nodes cluster together), similarity (which nodes resemble each other), pathfinding, and node embeddings. GDS projects a subgraph into memory, runs the algorithm, and writes results back as node properties or new relationships.

While the workshop does not run GDS algorithms directly, the pattern matters conceptually. Graph-enriched search in Lab 4 traverses explicit relationships to gather context around a vector match. GDS extends this by computing implicit structure: two companies might not share a direct relationship, but a similarity algorithm can identify that they face the same risk factors and compete in the same product categories. These computed relationships become additional context for retrieval.

== AI-Powered Graph Tools

Between the database and the AI layer sits a set of tools for querying, exploring, and managing graph data.

**Browser** is a web-based query workbench. You write Cypher, execute it, and see results as interactive node-link visualizations or tabular output. Lab 1 uses Browser to run sample queries against the knowledge graph and verify that data loaded correctly.

**Bloom** provides visual graph exploration without requiring Cypher. Users search for entities by name, expand their neighborhoods, and apply styling rules to distinguish node types. Bloom is useful for pattern discovery: visually expanding a company's relationships can surface connections that a pre-written query would miss.

**NeoDash** turns Cypher queries into dashboard components: charts, tables, maps, and graph visualizations arranged in a shareable layout.

**Data Import** provides a visual interface for mapping CSV files to graph structures, defining which columns become nodes, which become properties, and which become relationships. The workshop's seed data (companies, products, risk factors, asset managers, and their interconnections) follows this pattern.

**GraphQL** exposes graph data through a generated GraphQL API, translating GraphQL queries into Cypher automatically. This is relevant for application developers who want typed, schema-driven access to the knowledge graph without writing Cypher directly.

== Graph-Powered AI

The top tier of the platform turns the knowledge graph into infrastructure for AI applications. This is where Neo4j's capabilities intersect most directly with the GenAI stack.

=== Document Intelligence

Document Intelligence converts unstructured documents into knowledge graph structures. The process parses documents, splits text into chunks, extracts entities and relationships, generates vector embeddings, and stores everything as connected graph elements. The workshop's knowledge graph demonstrates the output of this pipeline: SEC 10-K filings become Document nodes linked to Chunk nodes (carrying text and embeddings), which connect to extracted Company, Product, and RiskFactor entities through typed relationships. The NEXT_CHUNK relationship preserves document ordering, so retrieval can reconstruct passages in sequence.

=== Aura Agents

Aura Agents is a no-code agent builder integrated into the AuraDB console. It generates a GraphRAG agent from the graph's ontology: point it at a database, and it creates tools, prompts, and retrieval strategies grounded in the actual schema. Lab 2 uses Aura Agents to build and test a conversational agent over the SEC filing knowledge graph without writing any code. Agents built in Aura can be deployed with a single click to a hosted MCP Server endpoint, making them accessible to external AI clients.

=== Ontology

Neo4j treats data modeling as a first-class capability, not a preliminary step that disappears after import. The Ontology tool provides a visual modeling workspace with a repository of domain-specific templates and schema enforcement. The data model for this workshop (Company, Product, RiskFactor, AssetManager, Document, Chunk and their connecting relationships) is itself an ontology that defines what the graph can represent and what questions agents can answer.

=== Agentic Brain

Agentic Brain provides foundational memory services that let agents learn from their own reasoning. The https://github.com/neo4j-labs/agent-memory[Neo4j Agent Memory] library implements a graph-native memory system with three tiers. Short-term memory stores conversation history with semantic search, so an agent can recall what was discussed recently. Long-term memory persists facts, preferences, and entities using a structured data model (Person, Object, Location, Event, Organization) with automatic entity resolution that merges duplicate references through fuzzy and semantic matching. Reasoning memory captures tool usage patterns, decision traces, and task outcomes, allowing agents to learn from past successes and failures rather than starting from scratch on every interaction.

The architecture stores all three memory types in Neo4j, combining vector search for semantic recall with graph traversal for relational context. When an agent remembers a prior conversation about a specific company, it can also traverse to that company's risk factors, competitors, and filings without a separate lookup. The library integrates with multiple agent frameworks including Strands, LangChain, and Pydantic AI, and supports Amazon Bedrock embeddings for vector representation.

=== Agentic Ecosystem

Neo4j integrates with the broader AI framework landscape through MCP Servers, driver libraries, and framework connectors. The Neo4j MCP Server exposes graph query capabilities over the Model Context Protocol, allowing any MCP-compatible agent to discover and call graph tools. Labs 4 and 6 connect to Neo4j MCP Servers hosted on AWS AgentCore, where agents call tools like `read-cypher` to execute Cypher against the knowledge graph. Driver libraries for Python, Java, JavaScript, Go, and .NET provide direct programmatic access. Connectors for LangChain, LlamaIndex, and the Strands Agents SDK let framework-native code interact with Neo4j without manual driver management.

== How the Platform Layers Connect

The three tiers form a stack where each layer builds on the one below:

----
Graph-Powered AI              AI-Powered Tools         Database & Analytics
(agent services)              (query + explore)        (store + compute)
      |                             |                        |
  Aura Agents ---+            Browser (Cypher)          AuraDB
  MCP Server  ---+            Bloom (visual)              |
      |          |            NeoDash (dashboard)     Vector Index
  Document    ---+                  |                 Fulltext Index
  Intelligence   |                  |                     |
      |          |                  |                  GDS (algorithms)
  Agentic     ---+                  |                     |
  Brain (memory) |                  |                     |
      |          +------------------+---------------------+
      |                             |
   Ontology  --- schema governs --> Knowledge Graph
                                    (Company, Product, RiskFactor,
                                     Document, Chunk, AssetManager)
----

The database stores the knowledge graph and maintains its indexes. The tools provide interactive access for developers and analysts. The AI layer exposes the graph to agents, converts documents into graph structures, and maintains the ontology that governs what the graph contains. All three tiers share the same underlying data and schema.
