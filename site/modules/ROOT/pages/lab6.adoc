== Embeddings and Vector Search

Lab 4 used pre-loaded embeddings from the setup seed data. This lab generates them from scratch. Amazon Bedrock Nova generates 1024-dimensional embeddings for each chunk of SEC filing text. Those vectors are stored as properties on Chunk nodes in Neo4j and indexed with a vector index (`chunkEmbeddings`) that supports approximate nearest-neighbor lookup. When a retriever receives a question, it embeds the question with the same model, queries the index for the closest chunk vectors, and returns the matched text.

.How vector search powers RAG
[%collapsible]
====
Vector search is the mechanism that makes Retrieval-Augmented Generation work. The system embeds a question, searches for the closest chunks in vector space, and feeds those chunks to the LLM as context. The LLM generates its answer grounded in the retrieved text rather than relying solely on its training data. The quality of retrieval determines the quality of generation; if vector search surfaces irrelevant chunks, the LLM produces an answer grounded in the wrong context.
====

== Chunking

SEC 10-K filings are too long to embed as single documents, so each filing is split into smaller passages. Each chunk becomes a Chunk node in the graph, linked to its source Document via `FROM_DOCUMENT` and to adjacent chunks via `NEXT_CHUNK` to preserve reading order.

.Chunking trade-offs
[%collapsible]
====
Larger chunks give the LLM more context during retrieval, which helps when the answer spans multiple sentences. Smaller chunks produce more precise results, returning only the relevant passage rather than a block of mixed content. A moderate size (500-1000 characters) is a reasonable starting point. The workshop uses `FixedSizeSplitter` from the neo4j-graphrag library, which splits text at fixed character boundaries. Tuning chunk size and overlap is an empirical process: adjust based on whether retrieved results contain too much noise or too little context.
====

== From Documents to Knowledge Graph

The pipeline builds both the structured and unstructured layers of the knowledge graph, then connects them:

1. **Load filing text as Document and Chunk nodes.** Each 10-K filing becomes a Document node. The filing text is split into chunks, each stored as a Chunk node linked to its source via FROM_DOCUMENT and to adjacent chunks via NEXT_CHUNK.
2. **Create structured entities.** Company, Product, and RiskFactor nodes are created from the structured data in the filing, linked by OFFERS, FACES_RISK, and FILED relationships.
3. **Generate embeddings on every Chunk.** Bedrock Nova embeds each chunk's text into a 1024-dimensional vector, stored as a property on the Chunk node. A vector index (`chunkEmbeddings`) is created over these embeddings.

.Entity extraction and resolution
[%collapsible]
====
The workshop's knowledge graph uses pre-built structured entities from JSON data rather than LLM-based entity extraction. The Company, Product, and RiskFactor nodes exist because they were loaded directly from structured sources, not because an LLM extracted them from filing text.

In a full production pipeline, an LLM reads each chunk and extracts entities mentioned in the text: company names, product references, risk factors. Entity resolution then deduplicates these, merging "NVIDIA Corporation," "Nvidia," and "NVDA" into a single Company node. Cross-linking connects extracted entities to existing graph structure, so a risk factor mentioned in filing text links directly to the RiskFactor node that other companies also reference.

The workshop sidesteps extraction because the SEC dataset already provides clean structured data. The conceptual model is the same: chunks connect to entities, entities connect to each other, and retrieval traverses both layers. Whether entities arrive via JSON loading or LLM extraction, the graph structure and retrieval patterns are identical.
====

== The Pipeline

The three notebooks form a single arc: create the graph, generate embeddings, then run graph-enriched retrieval queries.

Notebooks 01 and 02 build the foundation. Notebook 01 loads SEC filing text as Document and Chunk nodes linked by `FROM_DOCUMENT` and `NEXT_CHUNK` relationships, and creates the structured layer (Company, Product, RiskFactor) with cross-links (`FILED`, `FROM_CHUNK`). Notebook 02 generates vector embeddings on every Chunk node and creates a `chunkEmbeddings` vector index. Notebook 03 queries the graph using `VectorCypherRetriever`, which combines vector similarity search with Cypher graph traversal to enrich results with connected entities.

== VectorCypherRetriever

The `VectorCypherRetriever` adds a graph traversal step after vector search. It finds semantically similar chunks the same way `VectorRetriever` does, then executes a custom Cypher `retrieval_query` that traverses from each matched chunk to related entities in the graph. The chunk acts as an anchor: vector search determines relevance, then Cypher gathers the structured context around it.

A typical traversal follows FROM_DOCUMENT to the source Document, then FILED back to the Company, and from there to OFFERS, FACES_RISK, or other relationships. The LLM receives both the chunk text and the connected entities.

.How the retrieval_query parameter works
[%collapsible]
====
The `VectorCypherRetriever` accepts a `retrieval_query` parameter: a Cypher query that executes once per matched chunk. The retriever binds the matched Chunk node to a variable (`node`), and the retrieval query traverses from there to gather context. A typical query follows the chain from chunk to document to company:

[source,cypher]
----
MATCH (node)-[:FROM_DOCUMENT]->(d:Document)<-[:FILED]-(c:Company)
OPTIONAL MATCH (c)-[:FACES_RISK]->(r:RiskFactor)
RETURN node.text AS text, d.source AS document, c.name AS company,
       collect(r.name) AS riskFactors
----

This query runs for each chunk that vector search returns. The retriever collects the results and passes them to the LLM as enriched context. The design separates the search step (which chunk is relevant) from the enrichment step (what structured context surrounds it).
====

.The chunk-as-anchor constraint
[%collapsible]
====
Graph traversal starts from what vector search found. If the question does not surface relevant chunks, no amount of traversal compensates. A question targeting a specific entity by name ("How many products does NVIDIA offer?") may return better results with a direct Cypher query than with vector search, because the answer depends on counting relationships, not finding semantically similar text. VectorCypherRetriever is strongest when the question combines semantic relevance with structural context: the vector search finds the right passage, and the graph traversal enriches it with the entities and relationships that surround it.
====
