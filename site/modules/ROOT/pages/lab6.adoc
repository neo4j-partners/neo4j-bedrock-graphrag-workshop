== Model Context Protocol (MCP)

The Model Context Protocol is an open standard that defines how AI agents discover and interact with external tools and data sources. It provides a uniform interface: any MCP-compatible agent connects to any MCP-compatible server without custom integration code.

=== Three-Component Architecture

MCP defines a three-component architecture:

image::lab7-mcp-agent-architecture.png[MCP Agent Architecture,700]

**AI Agent (Client)**: The LLM-powered application that needs to access external data. It discovers available tools by asking the MCP server what operations are supported.

**MCP Server**: A lightweight service that exposes data source operations as tools. It translates between the MCP protocol and the data source's native API.

**Data Source**: The underlying system (Neo4j, a REST API, a file system, etc.) that holds the data.

=== Transport Options

MCP supports multiple transport mechanisms:

[cols="1,2,2"]
|===
| Transport | Description | Use Case

| **stdio**
| Standard input/output streams
| Local development, CLI tools

| **Streamable HTTP**
| HTTP-based streaming protocol
| Cloud deployments, remote servers
|===

This lab uses Streamable HTTP to connect to an MCP Gateway running on AWS.

=== Neo4j MCP Server Tools

The Neo4j MCP Server exposes two primary tools:

[cols="1,3"]
|===
| Tool | Description

| **get-neo4j-schema**
| Returns the graph schema: node labels, relationship types, and properties

| **read-neo4j-cypher**
| Executes a read-only Cypher query and returns results
|===

The agent discovers these tools automatically through the MCP protocol. It does not need to know about them in advance.

== AWS Deployment Architecture

The MCP infrastructure is pre-deployed on AWS:

----
Agent (Notebook)  -->  AgentCore Gateway (HTTPS)  -->  Neo4j MCP Server (Container)  -->  Secrets Manager  -->  Neo4j Aura
----

**Agent (Notebook)**: Your Jupyter notebook running a LangGraph or Strands agent. It connects to the gateway using a URL and access token from `CONFIG.txt`.

**AgentCore Gateway**: An AWS-managed HTTPS endpoint that authenticates requests and routes them to the MCP server.

**Neo4j MCP Server**: A containerized service that implements the MCP protocol and translates tool calls into Neo4j operations.

**Secrets Manager**: Stores the Neo4j Aura connection credentials securely. The MCP server retrieves them at runtime.

**Neo4j Aura**: Your graph database containing SEC financial data.

You do not need to set up any of this infrastructure. The gateway URL and access token are provided in `CONFIG.txt`.

== Schema-First Approach

Both notebooks instruct the agent to follow a schema-first pattern:

1. **Retrieve the schema** before writing any Cypher query. The agent calls `get-neo4j-schema` to learn what node labels, relationship types, and properties exist in the database.
2. **Write a Cypher query** based on the actual schema. This prevents the agent from guessing at property names or relationship types that do not exist.
3. **Execute the query** using `read-neo4j-cypher` and return the results.

This approach produces more accurate queries because the agent works with the real structure of the data rather than assumptions.

.How schema-first prevents query failures
[%collapsible]
====
Without the schema step, an LLM might generate `MATCH (c:Corp)-[:HAS_PRODUCT]->(p:Product)` when the actual label is `Company` and the relationship type is `OFFERS`. The query executes without error but returns zero results because `Corp` and `HAS_PRODUCT` do not exist in the graph. The agent would then report "no data found" as a confident answer.

With schema retrieval, the agent sees the actual labels (`Company`, `Product`, `RiskFactor`, `AssetManager`) and relationship types (`OFFERS`, `FACES_RISK`, `COMPETES_WITH`, `PARTNERS_WITH`, `OWNS`, `FILED`, `FROM_DOCUMENT`) before writing any query. The generated Cypher uses the correct vocabulary, and empty results genuinely mean no matching data rather than a query targeting non-existent schema elements.
====

== Lab 4 vs Lab 6

Lab 4 used MCP to execute pre-written Cypher queries. The retrieval logic was defined in advance, and MCP served as the transport layer between the agent and Neo4j. The agent selected which query to run, but did not write the queries itself.

Lab 6 removes that constraint. The agent inspects the schema, reasons about the question, writes its own Cypher, and executes it. This gives the agent full autonomy over how it queries the graph. The trade-off is that the agent's Cypher quality depends on the LLM's ability to reason about graph structures, which is why the schema-first approach is important.

== Why a Specialized Graph Agent

An agent that handles both SQL analytics and graph traversal in a single prompt must hold two query languages, two data models, and two sets of conventions in context simultaneously. SQL operates on rows, filters, and aggregations. Cypher operates on paths, patterns, and traversals. Mixing both dilutes the agent's focus: it may generate Cypher with SQL idioms (using WHERE clauses where pattern matching would be more precise) or SQL with graph assumptions (expecting JOIN-free traversals that relational databases cannot provide).

The architecture in this workshop separates concerns. The MCP agent in this lab handles only graph queries. It inspects the Neo4j schema, writes Cypher, and executes it. It does not need to know about SQL tables, aggregation functions, or relational joins. That focus produces better Cypher because the agent's entire context is graph-shaped.

In production systems, a supervisor or router agent sits above specialized agents and directs questions to the right one based on the data shape the question targets. Relationship traversals go to the graph agent. Time-series aggregations go to a SQL agent. Questions that span both get decomposed into sub-tasks routed to each specialist.

== Two Framework Options

This lab provides two notebook implementations. Both connect to the same MCP server and produce the same results. Choose the one that fits your interest.

**Strands** is the recommended option for this workshop. It is the same AWS-native SDK introduced in Lab 3, with built-in MCP support and a simpler API. Strands requires fewer lines of code to set up and is a good fit when you want a straightforward agent without the overhead of the LangChain ecosystem.

**LangGraph** is the full-featured option. It uses LangChain MCP adapters to load tools from the MCP server and provides fine-grained control over the agent loop. LangGraph is better suited for complex, multi-step workflows where you need to customize how the agent reasons, retries, or chains operations. This option is included for participants with prior LangGraph experience who want to compare the two approaches.
