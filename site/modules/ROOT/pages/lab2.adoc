.**At a Glance**

* **Aura Agents** — combine graph traversal and natural language queries in one conversational interface
* **"Create with AI"** — analyzes your graph schema and auto-generates tools with no Cypher to write manually
* **Two retrieval patterns** — Cypher Templates (deterministic traversals) and Text2Cypher (ad-hoc natural language queries)
* **Optional deployment** — expose the agent as a REST API endpoint or MCP server for integration with external clients

== What Are Aura Agents?

Aura Agents is Neo4j's no-code platform for building conversational AI interfaces grounded in graph data. Rather than writing retrieval logic by hand, you describe a use case in natural language and the platform generates an agent with tools tailored to your graph's schema. The agent then uses those tools to answer questions by combining graph traversals and dynamically generated Cypher queries.

The "Create with AI" workflow inspects the node labels, relationship types, and indexes in your knowledge graph and produces a fully configured agent. For the SEC financial dataset, this means the agent automatically understands how to look up company details, traverse ownership chains, compare risk factors, and search filing text, all without manual configuration.

== Two Retrieval Patterns

The generated agent uses two categories of tools, each suited to different question types.

=== Cypher Templates

Cypher Templates are pre-built graph traversal patterns mapped to specific questions. Each template targets a known pattern in the graph (company details, risk factors, ownership relationships) and executes a precise, deterministic Cypher query. Because the query structure is fixed, results are predictable and fast.

.Example
[%collapsible]
====
"What products does NVIDIA Corporation offer?" triggers a template that traverses `(Company)-[:OFFERS]->(Product)` and returns the product list. No query generation is needed. The template maps directly to the graph pattern.
====

=== Text2Cypher

Text2Cypher translates arbitrary natural language questions into Cypher queries at runtime. This is the most flexible retrieval pattern, able to answer questions the template authors did not anticipate, but it is also the least deterministic, since the generated query depends on the LLM's interpretation.

.Example
[%collapsible]
====
"How many products does NVIDIA Corporation mention?" generates a Cypher query like `MATCH (c:Company {name: 'NVIDIA Corporation'})-[:OFFERS]->(p:Product) RETURN count(p)` and executes it against the graph.
====

== Tool Type Comparison

[cols="1,2,2,1"]
|===
| Tool Type | Purpose | Best For | Reliability

| **Cypher Templates**
| Controlled, precise graph traversals
| Specific lookups, comparisons
| Deterministic

| **Text2Cypher**
| Flexible natural language to Cypher
| Ad-hoc questions about the data
| Variable
|===

== Tool Reliability

Templates and Text2Cypher sit at opposite ends of a reliability spectrum. Templates always return the same result for the same input but only cover questions someone anticipated. Text2Cypher handles everything else at the cost of predictability. The agent uses both: templates for known patterns, Text2Cypher for the long tail.

.Why Text2Cypher needs cross-checking
[%collapsible]
====
Because Text2Cypher generates a different query each invocation, subtle variations can produce incorrect answers silently. A query that filters on `"cybersecurity"` instead of `"Cybersecurity Threats"` returns no results for case-sensitive property values, and the agent may present that empty result confidently rather than flagging a possible query issue.

Running the same question twice and comparing results is a practical way to spot this variability.
====
